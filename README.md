# Lucille 3

Lucille 3 is a local-first AI productivity intelligence MVP. It observes explicit, privacy-bounded work evidence, turns it into a weekly efficiency report, and gives organisations a lightweight way to track where AI assistance can reduce repeated administrative work.

## Project Spine

Lucille now has the smallest runnable Node.js spine:

- `package.json` with build, syntax-check, and test scripts
- a plain JavaScript CLI at `src/cli.mjs`
- a build wrapper at `dist/cli.js` generated by `npm run build`
- deterministic mock analysis fixtures for tests and demos
- local Ollama visual provider plumbing for day-scoped raw media
- structured analysis artifacts under `storage/analysis/<DAY>/`
- weekly efficiency report generation under `src/reports/`
- model evaluation for hosted synthesis quality under `src/analysis/modelEvaluation.mjs`
- proposal validation and approval-gated skill export modules under `src/skills/`

`capture-once` now has a real, explicitly invoked ingestion path on macOS using the built-in `screencapture` command when Screen Recording permission is available. It writes one day-scoped raw frame and one strict `observation.v1` JSONL record. Long-running capture controls still only persist visible lifecycle state; there is no hidden background capture.

`capture-permission` requests and verifies macOS Screen Recording access before a real capture run. It uses the CoreGraphics permission prompt when available, verifies the actual `screencapture` path with a temporary probe frame, deletes that probe immediately, and opens System Settings to Privacy & Security > Screen Recording when manual approval is still required.

## RALF Loop

Run one iteration:

```bash
./scripts/run-ralf-loop.sh 1
```

Run the default v1 hardening loop:

```bash
./scripts/run-ralf-loop.sh 8
```

Useful overrides:

```bash
PROMPT_FILE=prompts/v1-hardening-ralf.md ./scripts/run-ralf-loop.sh 3
CODEX_BIN=codex LOG_DIR=logs/ralf ./scripts/run-ralf-loop.sh 5
```

The loop writes logs to `logs/ralf/` and runs available checks after each iteration.

## Make Targets

The everyday Makefile workflow is intentionally small:

```bash
make capture
make analyse
```

Both commands build the CLI wrapper first. `DAY`, `MODEL`, `PROVIDER`, `OPENAI`, and related options are still available as Make variables when needed.
The CLI automatically loads a repo-local `.env` file before running commands, so hosted synthesis and model evaluation can use `OPENAI_API_KEY` without exporting it in the shell.

Defaults:

- `DAY`: today
- `CAPTURE_INTERVAL`: `3` seconds
- `MODEL`: `moondream:1.8b`
- `PROVIDER`: `auto`
- `ANALYSE_LIMIT`: unset, meaning all observations
- `ANALYSE_OFFSET`: `0`
- `OPENAI_MODEL`: `gpt-5.5`
- `EVAL_MODELS`: `gpt-5.5`
- `REASONING_EFFORT`: `high`
- `DELETE_RAW_MEDIA`: `0`
- `APPROVE_EXPORT`: `0`

`make capture` runs in the foreground and captures one visible frame every 3 seconds. Stop it with Ctrl-C. Override the interval with `CAPTURE_INTERVAL=5 make capture`. On macOS each frame uses `screencapture -x` only as a visible operator command. If Screen Recording permission is unavailable, Lucille asks macOS for permission, probes the same `screencapture` path with a temporary frame, deletes that probe immediately, and opens System Settings when manual approval is needed. macOS still requires the operator to grant permission to the app running Lucille, such as Codex, Terminal, iTerm, or VS Code, and that app may need to be quit and reopened after approval.

Captured raw media is stored under:

- `storage/captures/<DAY>/raw-media/`

Every accepted frame appends one `observation.v1` record to:

- `storage/captures/<DAY>/observations.jsonl`

Persisted observations include only bounded structured metadata: observation id, timestamp, app/window hints when available, hostname-only domain, visible summary, redacted signals, and evidence ids. Excluded apps and domains are enforced before the screenshot command runs; excluded captures write only `observation-exclusions.jsonl` and do not create day-scoped raw media.

Capture-once treats raw media and observations as a single accepted frame. If `screencapture` or an injected capture command fails after creating a partial image file, Lucille deletes that partial raw media and does not append an observation.

`make analyse` defaults to `PROVIDER=auto` and `MODEL=moondream:1.8b`. In auto mode Lucille uses local Ollama for real captured observations with day-scoped raw media. If a real capture exists but raw media or Ollama is unavailable, analysis fails clearly instead of falling back to mock evidence. Mock analysis is used only for fixture-backed runs without captured observations, or when `PROVIDER=mock` is selected explicitly for deterministic testing. Explicit `PROVIDER=ollama` fails clearly if `storage/captures/<DAY>/raw-media/<OBSERVATION_ID>.<png|jpg|jpeg|webp>` is missing or Ollama is not reachable. Analysis writes:

For local vision testing, analyse a small chunk first:

```bash
make analyse DAY=2026-06-01 PROVIDER=ollama ANALYSE_LIMIT=5
make analyse DAY=2026-06-01 PROVIDER=ollama ANALYSE_LIMIT=5 ANALYSE_OFFSET=5
```

For the heavier local Llama vision model installed through Ollama:

```bash
make analyse DAY=2026-06-01 PROVIDER=ollama MODEL=llama3.2-vision:latest ANALYSE_LIMIT=5
```

- `storage/analysis/<DAY>/frame-analysis.jsonl`
- `storage/analysis/<DAY>/work-patterns.json`
- `storage/analysis/<DAY>/skill-proposals.json`

`make report DAY=<DAY>` reads those structured artifacts and writes a Markdown report to:

- `output/reports/<DAY>.md`

Reports include frame counts, provider/model details, capture surfaces, raw media lifecycle counts, work patterns, skill proposals, and privacy notes. They do not include raw screenshots, raw media paths, full URLs with query strings, raw document bodies, or raw message bodies.

`make model-eval DAY=<DAY>` compares candidate OpenAI models against the Lucille weekly efficiency report task using the same redacted structured evidence. It writes:

- `storage/analysis/<DAY>/model-evaluation.json`

The evaluator scores whether each model produces valid recommendations, cites evidence, quantifies weekly time savings, and includes organisation-level AI transformation metrics. It requires `OPENAI_API_KEY` and sends no raw screenshots or raw media paths.

Hosted synthesis now gates GPT output against a release-ready skill portfolio. Accepted proposal sets must include concrete skills across employee weekly reporting, workflow automation, AI assistance, manager monitoring, and enterprise rollout, each with owner, implementation steps, prerequisites, rollout metric, evidence, confidence, and estimated weekly savings.

When day-scoped `storage/captures/<DAY>/observations.jsonl` files exist, analysis accepts only the `observation.v1` schema used by the mock fixture: bounded visible summaries, hostname-only domains, redacted signals, evidence ids, and no extra persisted fields such as media paths or raw bodies.
Analysis also re-applies the excluded app and domain policy before provider selection, so manually written or stale observations from excluded surfaces fail before they can reach Ollama or optional OpenAI synthesis.

The Ollama path calls only a local model service, defaulting to `http://127.0.0.1:11434/api/generate`. It sends local raw media for the matching observation id to that local endpoint and persists only normalized structured `frame-analysis.v1` output. The top-level frame `evidenceId` is the observation raw-frame evidence id, for example `obs-...-raw-frame`, so downstream reports and skill proposals cite screenshot-backed evidence rather than generated mock counters. Raw screenshots are not sent to OpenAI by default.

`OPENAI=1` enables the optional hosted synthesis slice. It requires `OPENAI_API_KEY`, uses the OpenAI Responses API, and sends redacted structured frame/storyboard evidence only. Raw screenshots and raw media paths are not sent by default. The hosted response is normalized back into local `work-patterns.json` and `skill-proposals.json` with evidence IDs, confidence, and proposed-only export plans for Claude, Codex, Cursor, and ChatGPT.

By default, analysis keeps source images under `storage/captures/<DAY>/raw-media/` so operators can inspect or rerun model analysis. Set `DELETE_RAW_MEDIA=1` or pass `--delete-raw-media` only when you explicitly want analysis to delete day-scoped image files after structured analysis.

The deterministic no-paid-services happy path is:

```bash
make capture
make analyse
```

In a headless or permission-limited environment, skip `make capture`; `make analyse` falls back to readable fixtures in `fixtures/mock-observations.json`.

## Skill Export

Lucille writes at least three local skill proposals for each analysed pattern: a weekly report skill, a workflow automation queue, and a drafting/assistance skill. Hosted synthesis can produce a broader portfolio, but the local path is enough for an operator to review and generate useful skill bundles.

Start the local web UI to edit proposals, generate tool-specific skill files, and download a JSON bundle containing the Claude, Codex, Cursor, and ChatGPT artifacts:

```bash
make ui DAY=2026-06-01
```

Open the printed local URL. The UI reads and writes:

- `storage/analysis/<DAY>/skill-proposals.json`
- `output/skills/<DAY>/<PROPOSAL_ID>/`

Skill export is approval-gated. Preview planned files without writing tool-specific output:

```bash
node dist/cli.js export --day 2026-05-30 --proposal-id skill-id
```

Write the approved bundle explicitly:

```bash
node dist/cli.js export --day 2026-05-30 --proposal-id skill-id --approve-export
```

Approved exports are written under `output/skills/<DAY>/<PROPOSAL_ID>/`:

- Claude: `claude/SKILL.md`
- Codex: `codex/SKILL.md`
- Cursor: `cursor/.cursor/rules/*.mdc`
- ChatGPT: `chatgpt/instructions.md`, `chatgpt/knowledge.md`, and `chatgpt/actions.json`

The exporter reads only proposed `skill-proposals.json` records, validates evidence IDs and confidence, and does not write any tool-specific files unless approval is explicit.

## Safety Defaults

The RALF prompt instructs Codex to avoid destructive git commands, keystroke logging, clipboard capture, hidden capture, cookies, tokens, full URLs with query strings, raw document bodies, and raw message bodies. The current persisted observations, capture state, and analysis outputs pass a privacy validator that rejects forbidden field names and query-bearing URLs. Hosted OpenAI analysis must be explicit and should operate on redacted structured evidence rather than raw screenshots by default.

Capture lifecycle state is written to `storage/capture-state.json` by `src/capture/controller.mjs`. Saved state is read back through a strict schema validator before it is trusted, so unknown lifecycle fields, invalid statuses, weakened control flags, and full URLs in excluded domains are rejected. The capture controller records only operator-visible controls and policy metadata:

- start, pause, resume, stop, once, and status actions
- transition-guarded pause, resume, and stop controls that leave state unchanged when capture is not in the matching state
- visible controls required
- no hidden background capture
- real screen capture only for the foreground `make capture` loop or explicit `capture once`
- excluded apps and domains included in policy state
- no keystrokes, clipboard, audio, passwords, cookies, authentication tokens, raw document bodies, or raw message bodies
- source raw media retained by default after analysis unless deletion is explicitly requested

When raw media files exist under a day-scoped `raw-media` capture directory, `src/capture/rawMediaLifecycle.mjs` keeps known image file types after local analysis unless `--delete-raw-media` is explicitly enabled. The generated `work-patterns.json` records counts and retention policy only, not raw media paths or filenames.
